# AHP-App 구축 점검 보고서

**작성일:** 2026-07-13
**대상:** ahp-app (ahp-app.dreamitbiz.com) — AHP 다기준 의사결정 분석 SaaS 상용 제품판
**방법:** Supabase Management API 프로덕션 실측 + 익명 anon key 실호출 + 코드 전수 추적
**검증 원칙:** 모든 DDL은 트랜잭션 + ROLLBACK 으로 사전 검증 후 적용, 적용 후 실측 재확인

| 리포 | 역할 | 공개 | 도메인 |

## 2. 코드 이식 — 재작성하지 않은 이유

**백지 재작성을 하지 않았다.** ahp-basic 에 쌓인 것:

| | 규모 |
|---|---|
| 제품 코드 | 31,575줄 |
| 테스트 코드 | 5,113줄 (412건) |
| 커밋 | 405개 (2026-02-22 ~ 07-12, 5개월) |
| 검증된 계산 엔진 | 1,589줄 (AHP Saaty 표준 · 통계 10종 정확값) |

> **진짜 자산은 코드가 아니라 "검증된 정확성"이다.**
> ahp-basic 에서 통계 엔진의 `regularizedBeta` 결함으로 t·F·ANOVA·상관·회귀의 **p값이 전부 틀렸고,
> 그게 몇 달간 잠복하다 실사용에서야 드러났다.** 백지에서 다시 쓰면 이런 버그를 처음부터 다시 만난다.
> 이미 네 번째 재작성(Node → Django → ahp-basic → ahp-app)이 될 참이었다.

### 이식 내역

- **테이블 참조 119건 · RPC 참조 42건**을 소스 45개 파일에서 `ahp_` 접두어로 전환
- 테스트의 기대값(`toHaveBeenCalledWith`)·mock 분기(`table === 'x'`)·객체 키까지 동기화
- 브랜딩 43건(19파일) + OG·canonical·sitemap·robots.txt 도메인 전환
- 기능 영향 있던 것: `sessionStorage` 장바구니 키(`ahp_basic_cart` → `ahp_app_cart`), 결제 팝업 `siteSlug`

**전환 중 오탐 1건 발견·교정:** `p.type === 'criteria'` 는 페이지 타입이지 테이블명이 아니라서 되돌렸다.

## 3. DB 스키마 — 실측 기반 구축

### 왜 마이그레이션을 재생하지 않았나

ahp-basic 의 마이그레이션은 **39개 파일 3,318줄이 서로를 덮어쓰며 쌓인 구조**다.
그대로 재생하면 현재 프로덕션 스키마와 어긋난다.
→ **"지금 살아 있는 스키마"를 Management API로 실측 추출**해 원본으로 삼았다.

### 코드 스캔이 놓친 테이블 5개 발견

코드의 `.from()` 호출만 스캔하면 19개 테이블이 나온다. 그런데 **RPC 함수 본문을 파싱**하니
RPC로만 접근하는 테이블 5개가 더 있었다:

```
community_posts   page_views   plan_prices   point_transactions   project_plans
```

→ 총 **24개 테이블**. 이걸 놓쳤으면 스키마가 깨졌다.

### 구축 결과

| 객체 | 수량 |
|---|---|
| 테이블 | 24 |
| 컬럼 | 236 |
| 제약조건 | 85 (PK 19 · FK 30+1 · UNIQUE 10 · CHECK 10) |
| 함수 | 44 (RPC 33 + RLS 헬퍼 + 트리거 함수) |
| RLS 정책 | 110 |
| 트리거 | 4 + 가입 트리거 1 |

**검증 절차:** 전체 스키마를 `BEGIN … ROLLBACK` 트랜잭션으로 사전 실행해 문법·참조 정합성을 확인한
뒤 적용했다. 그 과정에서 3건의 오류를 사전에 잡았다:

1. `ARRAY` 타입 — `information_schema` 가 `uuid[]` 를 `ARRAY` 로 보고 → `pg_attribute.format_type()` 으로 교체
2. 정책명에 공백 포함(`Anyone can insert lecture applications`) → 따옴표로 감쌈
3. 함수 간 전방 참조(`is_superadmin`) → `SET LOCAL check_function_bodies = off` (pg_dump 표준 방식)

## 4. ahp-basic 대비 교정한 결함 3건

### ① 평가 데이터 FK 누락 → 고아 데이터 (가장 위험했던 것)

ahp-basic 의 `pairwise_comparisons` 는:

```sql
criterion_id UUID,     -- ← FK 제약 없음!
row_id UUID NOT NULL,  -- ← FK 없음
col_id UUID NOT NULL,  -- ← FK 없음
```

평가 데이터가 평가기준·대안의 ID를 가리키는데 **외래키가 없었다.**

> **평가기준을 수정·삭제하면, 이미 수집된 쌍대비교 데이터가 "존재하지 않는 ID를 가리키는 고아 행"이 된다.
> 에러도 안 나고 행도 그대로 남아서 겉보기엔 멀쩡한데 계산 결과만 조용히 틀린다.**
> CASCADE 삭제보다 위험하다 — 삭제는 눈에 보이지만 이건 안 보인다.

연구·논문용 도구에서 이건 치명적이다. **ahp-app 에서는 FK + `ON DELETE CASCADE` 로 원천 차단했다.**

> 단, `row_id`/`col_id` 는 criteria 와 alternatives 를 **둘 다** 가리킬 수 있어(기준 비교 vs 대안 비교)
> 단일 FK 를 걸 수 없다. 앱 레벨에서 지킨다. **향후 정규화 검토 대상.**

### ② 함수 8개 `search_path` 미고정

`assign_plan_to_project` · `check_user_status` · `create_community_post` · `delete_community_post` ·
`get_community_posts` · `grant_free_plan` · `increment_post_views` · `is_admin`

**전부 SECURITY DEFINER 인데 `search_path` 가 없었다** (search_path 주입 위험).
→ 새 스키마에서 전부 `SET search_path TO 'public'` 고정. **미고정 `ahp_` 함수: 0개** (실측 확인)

### ③ `user_profiles` INSERT 정책이 `WITH CHECK (true)`

**누구나 임의의 프로필 행을 삽입할 수 있었다.**
→ `WITH CHECK (auth.uid() = id)` 로 본인 행만 삽입 가능하게 교정.

### 추가: 권한 설계

ahp-basic 은 Supabase 기본값대로 `anon`/`authenticated` 에 **테이블 ALL 권한**이 열려 있었다
(`DELETE`·`TRUNCATE` 포함). RLS가 막고는 있지만, ahp-app 은 **최소 권한**으로 간다.

**⚠️ 컬럼 REVOKE 는 테이블 권한을 못 깎는다** — 역할이 테이블 레벨 `SELECT` 를 가지면 그것이 모든 컬럼을
덮으므로 `REVOKE SELECT (col)` 은 경고만 뜨고 무효다(ahp-basic 에서 실제로 이것 때문에 1차 수정이 실패했다).
→ **테이블 SELECT 를 주지 않고, 안전한 컬럼만 컬럼 레벨로 GRANT.**

`ahp_projects` 는 익명에게 **14개 컬럼만** 부여한다. 비밀 2개 제외:
`access_code`(설문 접근 비밀번호) · `result_share_token`(결과 공유 링크 토큰)

> 이 14개 목록은 코드의 `ProjectContext.PROJECT_FIELDS` 와 **정확히 일치해야 한다.**
> 어긋나면 익명 평가 화면이 42501 로 깨지거나 유출이 되살아난다.

## 5. 🔴 발견: 플랫폼 전체 위험 (미조치)

> **이건 ahp-app 문제가 아니다. 111개 사이트 전부의 문제다.**

`auth.users` 에 사이트별 가입 트리거가 **14개** 달려 있는데, 그중 **5개가 방어코드 없이 동작한다.**

| 트리거 함수 | EXCEPTION | search_path |
|---|:---:|:---:|
| `handle_agent_new_user` | ❌ | ❌ **둘 다 없음** |
| `instructor_handle_new_user` | ❌ | ✅ |
| `ppt_handle_new_user` | ❌ | ✅ |
| `rest05_handle_new_user` | ❌ | ✅ |
| `handle_plan_new_user` | ✅ | ❌ |

### 왜 위험한가

`auth.users` INSERT 시 **트리거 중 하나라도 예외를 던지면 INSERT 전체가 롤백된다.**
→ 그 사이트뿐 아니라 **111개 사이트 전부의 회원가입이 동시에 마비된다.**

**2026-06-19 에 실제로 이 사고가 났다.** 같은 구조가 지금 5개 남아 있다.

### 조치 스크립트 (준비 완료 · 미적용)

`supabase/schema/99_URGENT_signup_trigger_hardening.sql`

로직은 그대로 두고 **방어코드만 추가**한다:
- `SET search_path TO 'public'` (주입 방지)
- `EXCEPTION WHEN OTHERS THEN RETURN NEW;` (프로필 생성이 실패해도 가입은 진행)

**트랜잭션 + ROLLBACK 으로 문법 검증 완료.** `CREATE OR REPLACE` 라 멱등(중복 실행 안전).
위험도 매우 낮음(로직 불변).

> **공유 프로덕션이라 자동 실행하지 않았다.** SQL Editor 에서 적용 후 **실제 회원가입 1건을 테스트**할 것.

## 8. 작업 원칙 (지켜진 것)

- **공유 프로덕션 DDL은 사전 ROLLBACK 검증 → 적용 → 실측 재확인** 순으로만 진행했다.
- **가입 트리거 하드닝처럼 111개 사이트에 영향을 주는 변경은 자동 실행하지 않았다.**
- 점검용 관리 토큰은 사용 후 폐기됐다(현재 Unauthorized 확인).
- 모든 결함 주장은 **실측으로 확인**했다. 문서의 사실 주장은 코드·DB와 대조 검증했다.
