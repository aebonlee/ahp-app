## 1. 무엇이 문제였나

`auth.users` 에 사이트별 가입 트리거가 14개 달려 있다. 한 사람이 가입하면 14개가 전부 실행되어
각 사이트의 프로필 행을 만든다.

**PostgreSQL 트리거는 하나라도 예외를 던지면 원본 INSERT 가 통째로 롤백된다.**
즉 `ppt` 사이트의 프로필 테이블에 문제가 생기면, **ppt 뿐 아니라 111개 사이트 전부에서 가입이 안 된다.**

### 조치 전 상태 (2026-07-13 실측)

| 트리거 함수 | EXCEPTION | search_path |
|---|:---:|:---:|
| `handle_agent_new_user` | ❌ | ❌ **둘 다 없음** |
| `instructor_handle_new_user` | ❌ | ✅ |
| `ppt_handle_new_user` | ❌ | ✅ |
| `rest05_handle_new_user` | ❌ | ✅ |
| `handle_plan_new_user` | ✅ | ❌ |

(나머지 9개는 이미 방어코드를 갖추고 있었다)

---

## 2. 무엇을 바꿨나

**로직은 한 줄도 건드리지 않았다.** 추가한 것은 방어코드 2줄뿐이다.

```sql
 SET search_path TO 'public'                 -- search_path 주입 방지
...
EXCEPTION WHEN OTHERS THEN
  RETURN NEW;   -- 프로필 생성이 실패해도 가입은 진행
```

`EXCEPTION WHEN OTHERS THEN RETURN NEW` 가 핵심이다.
이제 트리거가 실패하면 **그 사이트의 프로필만 조용히 건너뛰고, 회원가입 자체는 성공**한다.
전 사이트 마비가 **구조적으로 불가능**해졌다.

---

## 3. 어떻게 안전하게 적용했나

111개 사이트의 회원가입이 걸린 작업이라 안전장치를 겹겹이 뒀다.

### 검증 도구: 가입 시뮬레이션

실제 계정을 만들지 않고 **진짜 가입 경로를 그대로 통과시키는** 방법을 썼다.

```sql
DO $sim$
DECLARE v_id uuid := gen_random_uuid(); v_cols text;
BEGIN
  -- 생성 컬럼(confirmed_at 등)은 제외하고 컬럼 목록을 만든다
  SELECT string_agg(quote_ident(attname), ', ' ORDER BY attnum) INTO v_cols
    FROM pg_attribute
   WHERE attrelid='auth.users'::regclass AND attnum>0
     AND NOT attisdropped AND attgenerated='';

  CREATE TEMP TABLE _sim ON COMMIT DROP AS SELECT * FROM auth.users LIMIT 1;
  UPDATE _sim SET id=v_id, email='trigger-sim-…@example.invalid', phone=NULL;

  EXECUTE format('INSERT INTO auth.users (%s) SELECT %s FROM _sim', v_cols, v_cols);
  --  ↑ 여기서 14개 트리거가 전부 발화한다

  RAISE EXCEPTION 'SIM_OK …';   -- 일부러 예외를 던져 전체 롤백
END $sim$;
```

`RAISE EXCEPTION` 으로 끝나므로 **테스트 계정도 프로필도 절대 남지 않는다.**
INSERT 가 성공하면 = 회원가입이 정상이라는 뜻이다.

> 처음 시도에서 두 번 막혔고 고쳤다:
> ① `SELECT *` 로는 안 된다 — `confirmed_at` 이 **생성 컬럼**이라 삽입 불가 → 생성 컬럼 제외
> ② 결과를 `SELECT` 로 받으면 API 가 마지막 구문(`ROLLBACK`) 결과만 돌려준다 → `RAISE EXCEPTION` 으로 메시지에 실어 회수

### 적용 절차

1. **함수 5개 정의 백업** → 즉시 복구용 원복 스크립트 생성
2. **작업 전 기준선** — 시뮬레이션으로 현재 가입이 정상임을 확인 (✅ 통과)
3. **함수를 하나씩** 적용 (5개를 한꺼번에 하지 않음)
   — 변경이 작은 것부터: `handle_plan`(search_path만) → … → `handle_agent`(둘 다 없음)
4. **매 함수마다 시뮬레이션 재실행** — 실패하면 즉시 그 함수만 백업본으로 원복하고 중단
5. 최종 검증

`CREATE OR REPLACE` 는 원자적이라 적용이 실패하면 아무것도 바뀌지 않는다.

### 적용 로그

```
▶ handle_plan_new_user        — search_path 고정              → ✅ 가입 정상
▶ rest05_handle_new_user      — EXCEPTION 추가                → ✅ 가입 정상
▶ ppt_handle_new_user         — EXCEPTION 추가                → ✅ 가입 정상
▶ instructor_handle_new_user  — EXCEPTION 추가                → ✅ 가입 정상
▶ handle_agent_new_user       — search_path + EXCEPTION 추가  → ✅ 가입 정상
```

**매 단계에서 가입이 정상이었다. 한 번도 중단하지 않았다.**

## 4. 최종 검증

### ① 트리거 14개 전부 방어코드 확보

```
✅ ahp_handle_new_user          ✅ bulid_handle_new_user      ✅ handle_agent_new_user
✅ handle_coding_new_user       ✅ handle_forjob_new_user     ✅ handle_new_user
✅ handle_plan_new_user         ✅ instructor_handle_new_user ✅ jdy_handle_new_user
✅ kdn29_handle_new_user        ✅ ppt_handle_new_user        ✅ rest05_handle_new_user
✅ tpl_handle_new_user          ✅ www_handle_new_member
```
**방어코드 없는 트리거: 0개**

### ② 🔑 핵심 — 트리거가 실패해도 가입이 되는가

말로만 "안전해졌다"고 하지 않고 **실제로 고장을 내서 확인**했다.

`ppt_profiles` 에 `CHECK (false) NOT VALID` 제약을 걸어 **`ppt_handle_new_user` 가 100% 실패하는 상황**을
만든 뒤 회원가입을 시도했다. (`NOT VALID` 라 기존 행은 검사하지 않고 신규 INSERT만 막는다)

```
✅✅ 가입 성공 — 정상사이트프로필=1 · 고장난ppt프로필=0
```

> **ppt 트리거가 100% 실패하는데도 회원가입이 완료됐다.**
> 고장난 사이트의 프로필만 건너뛰고(0건), 나머지 사이트는 정상 생성(1건).
> **수정 전이었다면 이 상황에서 111개 사이트 전체 가입이 마비됐다.**

### ③ 뒷정리 확인

| 검사 | 결과 |
|---|---|
| 테스트용 임시 제약(`_sim_fail`) 잔존 | ✅ 0건 |
| 테스트 계정(`trigger-sim-*`, `fail-test-*`) 잔존 | ✅ 0건 |
| 테스트 프로필 잔존 (user_profiles / ahp / ppt) | ✅ 0건 |
| 최종 가입 시뮬레이션 | ✅ 정상 |

**모든 테스트가 롤백되어 DB에 흔적이 남지 않았다.**
